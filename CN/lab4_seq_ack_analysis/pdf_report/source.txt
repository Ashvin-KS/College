Attack 1: TCP SYN Flood
AttributeDetail
Type Denial of Service (DoS)
SeverityHigh
Target Server's half-open connection table
Description
In a SYN Flood attack, the attacker sends a large number of SYN packets without completing
the three-way handshake. Each incomplete handshake occupies a slot in the server's backlog
queue. When the queue fills up, legitimate connection requests are dropped.
sudoaptinstall hping3 -y

How the Three-Way Handshake is Exploited
Command
Flag Meaning
-S Set SYN flag
-p 80 Target port 80 (HTTP)
-c 50 Send 50 packets then stop
Output
Observation: The server replies with SYN-ACK (flags=SA) for every SYN it receives,
confirming it is processing the half-open connections. At scale, this exhausts the server's
connection table.
Mitigation
Attacker          Server
|  -- SYN ---->   |  Server allocates resources
|  -- SYN ---->   |  Slot occupied
|  -- SYN ---->   |  Queue fills up ...
|  (no ACK)       |  Legitimate users blocked
sudo hping3 -S-p80-c50172.217.24.206
Enable SYN cookies (net.ipv4.tcp_syncookies = 1)
Rate-limit incoming SYN packets with iptables
Use a reverse proxy or load balancer with SYN flood protection

Attack 2: TCP RST (Reset) Attack
AttributeDetail
Type Connection Disruption
SeverityMedium–High
Target Active TCP sessions
Description
A TCP RST attack sends forged RST packets to forcefully close an active connection between
two parties. When a host receives a RST packet with a valid sequence number, the OS
immediately tears down the session without further negotiation.
Impact
Command
Flag Meaning
-R Set RST flag
-p 80 Target port 80
-c 20 Send 20 packets then stop
Output
Observation: The server drops all RST packets silently — 0 received. This is expected
behavior, as RST packets are not acknowledged in the TCP protocol.
Mitigation
Interrupts SSH, Telnet, VPN, video streams, and database connections
Can be used as a targeted disruption against specific sessions
sudo hping3 -R-p80-c20172.217.24.206

Attack 3: TCP Data Flood
AttributeDetail
Type Denial of Service (DoS) / Bandwidth Exhaustion
SeverityHigh
Target Server bandwidth and CPU
Description
Unlike a SYN flood, a data flood first completes the TCP handshake and then sends large
amounts of data to exhaust the server's bandwidth and CPU. The PUSH flag forces the server's
TCP stack to immediately deliver data to the application layer without buffering.
Command
Flag Meaning
-S SYN flag (establish connection)
-P PUSH flag (force immediate processing)
-p 80 Target port 80
-c 50 Send 50 packets
-d 100 100 bytes of payload per packet
Output
Data Flood – packets with 40 headers + 100 data bytes
Use encrypted protocols (TLS/SSH) to prevent sequence number sniffing
Implement stateful firewall inspection
Monitor for anomalous RST packet volumes in Wireshark
sudo hping3 -S-P-p80-c50-d100172.217.24.206

_Data Flood statistics – 50 sent, 50 received, avg RTT 140.1 ms
Observation: 100% packet loss was not observed because the server acknowledges the data
(it has a valid session). The average RTT of ~140 ms indicates server processing load.

Attack 5: FIN-ACK Flood
AttributeDetail
Type Denial of Service (DoS)
1. Attacker places themselves in the network path (ARP spoofing, promiscuous mode, etc.)
2. Wireshark captures the live traffic and reveals the current SEQ and ACK numbers
3. The attacker crafts packets with matching sequence numbers using hping3
4. The injected packets are accepted by the target as valid data
Out-of-order packets appearing in the capture
Duplicate ACKs from the legitimate receiver
Unexpected RST packets as the real client detects desynchronization
sudo hping3 -S-p80-M<sniffed_seq_num>-c10172.217.24.206
Use TLS/SSL for all sensitive communications
Implement IPSec for network-layer encryption
Use unpredictable, randomized sequence numbers (enabled by default in modern OS)
Monitor for duplicate ACKs and out-of-order segments in Wireshark

AttributeDetail
SeverityMedium
Target Server's connection teardown resources
Description
A FIN-ACK flood sends a large number of FIN-ACK packets to a target. FIN-ACK packets are
normally used to gracefully close connections. Processing these forces the server to allocate
resources for fake session teardowns, consuming CPU cycles and memory.
Command
Flag Meaning
-F Set FIN flag
-A Set ACK flag
-p 80 Target port 80
-c 50 Send 50 packets then stop
Output
Observation: 100% packet loss — the server drops all FIN-ACK packets silently because there
is no matching session state for these unsolicited packets. This confirms the target correctly
validates connection state before processing teardown requests.
Mitigation
sudo hping3 -F-A-p80-c50172.217.24.206
Stateful firewall rules that drop unsolicited FIN-ACK packets
Rate-limit connection teardown requests
Log and alert on FIN-ACK flood patterns in IDS/IPS

Attack 6: TCP Reflection Attack
AttributeDetail
Type Distributed Denial of Service (DDoS)
SeverityCritical
Target Victim's network bandwidth
Description
In a reflection attack, the attacker spoofs the victim's IP as the source address and sends SYN
packets to one or more reflector servers. The reflectors respond with SYN-ACK packets to the
victim — not the attacker — flooding the victim with unsolicited traffic. Using multiple reflector
servers turns this into a full-scale DDoS attack.
Attack Flow
Sample Command
Flag Meaning
-S SYN flag
--spoof <IP> Spoof the source IP address to the victim's address
-c 50 Send 50 packets
Wireshark Observation
The victim would see a flood of SYN-ACK packets arriving from the reflector. Since the victim
never sent the original SYN, its TCP stack will respond with RST packets, generating additional
outbound traffic as a side effect.
Attacker ---[SYN, src=victim_IP]---> Reflector 1 ---[SYN-ACK]---> Victim
Attacker ---[SYN, src=victim_IP]---> Reflector 2 ---[SYN-ACK]---> Victim
Attacker ---[SYN, src=victim_IP]---> Reflector N ---[SYN-ACK]---> Victim
|
RST flood (side
effect)

Conclusion
All six TCP-based attacks were successfully demonstrated using hping3 and observed in
Wireshark.
Key findings:
Ingress filtering (BCP 38 / RFC 2827) — ISPs block outbound packets with spoofed source
IPs
Anti-spoofing ACLs on edge routers
DDoS mitigation services
The SYN Flood and Data Flood produced measurable responses from the target server,
confirming that the server processes and allocates resources for each incoming packet.
The RST and FIN-ACK floods confirmed that unsolicited packets with no matching session
state are dropped silently — the server does not waste resources acknowledging them.

Wireshark's TCP stream analysis and sequence number tracking remain effective tools for
identifying and analyzing these attack patterns in real-time traffic.
